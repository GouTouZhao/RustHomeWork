## 3. 项目背景与目标
**项目背景：**
随着网络技术的发展，人们对于高性能和高并发交流平台的需求日益增加。传统的单体架构系统在用户量增长时往往面临性能瓶颈。本项目以现有的 `openim`（基于 Go 语言）项目为基础进行改进重构，全面采用内存安全且性能极高的 Rust 语言重写后端服务，同时剥离原有系统中繁杂的博客功能，专注于打造一个纯粹的、企业级的论坛交流平台。

**项目目标：**
- **功能目标**：实现完整的论坛发帖、浏览、评论、个人主页等核心功能。
- **技术目标**：
  - 将原有的 Go 后端微服务框架平滑迁移至 Rust 语言体系。
  - 维持原有的企业级微服务架构体系，包括使用 gRPC 进行服务间通信以及使用网关进行 API 路由转发。
  - 实现基于 Kubernetes (k8s) 的容器化高可用部署，支持通过 `docker push` 与 `k8s apply` 流程一键发布。
  - 前端基于 Vue 3 框架，构建响应式、轻量化的论坛用户界面。

## 4. 系统设计与实现思路
# 4. 系统设计与实现思路

**整体架构**
项目采用前后端分离的微服务架构，使用 Kubernetes 进行容器编排。
- **前端 (ForumFrontend)**：基于 Vue 3 + Vite，提供首页、分区页、文章详情页、发帖页和个人中心等页面，所有请求均经由 API Gateway 转发。
- **API Gateway**：使用 Rust 的 Axum 框架实现，负责跨域、统一路由与鉴权。
- **RPC 微服务层**：包括 `user`、`bmanager`、`admin`、`image`/`static_svc` 等服务，采用 gRPC + protobuf 定义接口，实现高效、强类型的服务间通信。
  - **`user` 微服务**：主要负责用户身份的认证与授权，包括注册、登录、个人信息管理等。它独立封装了对用户敏感数据的访问，确保了安全隔离。
  - **`bmanager` (业务管理) 微服务**：作为论坛业务的核心后台，专门处理帖子的发布、版块分类管理、内容存储交互及删改等操作。
  - **微服务架构的优势**：通过将单体应用拆分为如 `user`、`bmanager` 这样职责单一的独立微服务，系统实现了**业务解耦**（某个业务模块故障不会导致整站瘫痪，e.g: user挂了，不影响static（负责博客等静态资源的查看，用户无需登录）），并支持针对高并发模块（如帖子读取）进行**独立扩容**。此外，每个微服务可独立部署和演进，极大提升了项目的可用性与团队协作效率。

**关键数据结构与数据库设计**
- **帖子表结构 (MySQL)**：为支撑核心论坛业务，设计了完备的文章/帖子数据表结构。核心字段包含 `article_id`（主键）、`title`（标题）、`content`（富文本正文）、`author_id`（关联用户）、`category_id`（所属分区）、`cover_image`（首图链接）以及用于数据统计的 `view_count`、`like_count`、`comment_count` 等字段。该结构支持高效的并发读写与灵活的联表查询。
- 通过 protobuf 定义统一的数据模型，确保跨语言、跨服务的数据一致性。
- 统一的错误类型 `Error` 贯穿所有服务，便于全局拦截与日志记录。
- 引入 Redis 缓存热门帖子与用户信息，显著降低 MySQL 查询压力。

**部署设计**
- 每个微服务均提供 `Dockerfile`，使用轻量化基础镜像构建。
- `deployments/` 目录保存 K8s `yaml` 资源文件，配合 `start‑k8s.bat` 脚本实现“一键式”编译、镜像构建、部署。

**本项目与原项目的区别、修改和改进之处：**
1. **技术栈的底层重构**：将后端的 Go 语言全面替换为 Rust 语言。利用 Rust 的所有权机制和零成本抽象特性，在不增加内存开销的前提下，极大地提升了微服务的内存安全性和高并发处理能力。项目全面引入了 `tokio` 和 `axum` 等 Rust 高性能生态库。
2. **功能剥离与聚焦**：去除了原项目中冗余的“个人博客”等分区模块，专注实现“论坛与发帖”功能。前端 UI 层面基于 Vue 3 进行了针对性的精简和修改，使界面更契合纯粹的论坛需求。
3. **架构原样保留与生态适配**：在保持了原有企业级微服务封装、调用逻辑、报错标准化等优秀设计的前提下，成功用 Rust 语言走通了整个 k8s 和 gRPC 流程，展现了良好的架构可移植性。
